# MedGraph — Clinical Literature Knowledge Graph + RAG API

A knowledge graph built from medical research literature on diagnostic
imaging AI, extracted using Claude, stored in Postgres, served through a
FastAPI backend with a RAG-based Q&A endpoint, analyzed with NetworkX/pandas,
and deployed on AWS ECS Fargate.

**Live demo:** `http://54.224.95.137:8000` *(Fargate task IP — may change on
restart; see [Deployment](#deployment) to redeploy or check current status)*

```bash
curl http://54.224.95.137:8000/health
curl "http://54.224.95.137:8000/query?entity=glioma"
```

## What it does

1. Takes a set of medical research paper summaries (diabetic retinopathy,
   brain tumors, esophageal cancer, thyroid nodules — diagnostic imaging AI)
2. Uses Claude to extract structured entities (diseases, imaging modalities,
   techniques, anatomy) and the relationships between them
3. Stores the result as a graph in Postgres (nodes + edges tables)
4. Serves it through a FastAPI backend:
   - `GET /entities` — list all entities, optionally filtered by type
   - `GET /query?entity=X` — 1-hop subgraph around an entity, with source
     paper citations
   - `POST /ask` — ask a natural-language question; retrieves the relevant
     subgraph and has Claude answer using only those facts (RAG)
5. Provides a NetworkX/pandas analysis layer for graph algorithms (degree
   centrality, shortest path between entities) and a rendered visualization
6. Runs as a multi-container Docker stack, deployed on AWS ECS Fargate

## Run it locally

```bash
python3 -m venv venv && source venv/bin/activate
pip install -r api/requirements.txt anthropic networkx pandas matplotlib

export ANTHROPIC_API_KEY=sk-ant-...

# Extract the graph from the seed papers
python3 extract/extract_entities.py

# Bring up Postgres + API together
docker-compose up --build

# In another terminal, load the graph into the running database
export PGHOST=localhost PGPORT=5433 PGDATABASE=medgraph PGUSER=postgres PGPASSWORD=postgres
python3 api/load_graph.py

curl localhost:8000/entities
curl -X POST localhost:8000/ask -H "Content-Type: application/json" \
  -d '{"question": "What techniques detect diabetic retinopathy?"}'
```

Run the analysis layer separately:
```bash
export PGHOST=localhost PGPORT=5433
python3 analysis/graph_analysis.py
```

## Deployment

Deployed on AWS ECS Fargate as a two-container task (Postgres + API sharing
a network namespace, so the API reaches Postgres at plain `localhost`).

```bash
# Build for the Fargate architecture (x86_64), not your local Mac's ARM64
docker build --platform linux/amd64 -f api/Dockerfile -t medgraph-api .
docker tag medgraph-api:latest <ECR_URI>:latest
docker push <ECR_URI>:latest

aws ecs update-service --cluster medgraph-cluster \
  --service medgraph-api-service --force-new-deployment --region us-east-1
```

Full task definition in `ecs/task-definition.json`.

## Design notes / trade-offs

- **Postgres instead of a dedicated graph database (Neo4j/Neptune):** keeps
  the stack simpler — one database technology, no separate graph engine to
  run or pay for — while still modeling a true node/edge graph (traversal is
  a join). NetworkX is layered on top for graph *algorithms* (centrality,
  shortest path) that are awkward to express in SQL; Postgres remains the
  system of record.
- **RAG retrieval is keyword-based, not embeddings.** Fine at this graph's
  scale (dozens of nodes); at a larger scale, `retrieve_subgraph_text` in
  `main.py` would be swapped for vector similarity search.
- **Small seed corpus (12 papers) by design.** Proves the full pipeline
  (extraction → storage → API → RAG → analysis) end to end before scaling
  ingestion volume — swapping in a live PubMed ingestion script is a
  drop-in change to the first stage only; nothing downstream changes.
- **Fargate over EC2-based ECS:** no server management, pay only for the
  task's actual running time — appropriate for a project at this scale.

## Known limitations

- A handful of entities are mis-typed by the extraction step (e.g. some
  symptoms tagged as "Disease") — LLM extraction accuracy issue, not a
  pipeline bug; would tighten with a stricter type taxonomy or a validation pass.
- The Fargate task's public IP is not static — it changes if the service
  is redeployed or the task restarts. A production version would sit behind
  an Application Load Balancer with a stable DNS name.
- No automated tests yet.
